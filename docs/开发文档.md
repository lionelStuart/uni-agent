# 可插拔 Skills 通用 Agent Client 开发文档

## 1. 开发目标

本项目首期要交付的是一个 Agent Client，而不是平台服务。

首期交付物应满足：

- 用户可以通过 CLI 或本地接口发起任务。
- Client 可以加载本地 Skills。
- Agent 可以根据任务选择 Skill、调用工具并在沙盒中执行。
- 执行过程有日志、有结果、有错误信息。

## 2. 核心特性

### 2.1 Client 入口

- CLI 任务入口
- 本地配置加载
- 工作目录管理
- 会话管理

### 2.2 Agent Runtime

- 结构化计划生成
- 多步骤动作执行
- 执行观察与局部重规划
- 统一结果输出

### 2.3 Skills 插件系统

- 本地 Skills 扫描
- Skill Manifest 校验
- Skill 匹配与优先级解析
- 多 Skill 组合与冲突处理

### 2.4 工具系统

- 工具注册与发现
- 入参 schema 校验
- 工具权限分级
- 标准化调用结果

### 2.5 通用沙盒

- 受控 shell 执行
- 文件读写限制
- 工作目录白名单
- 超时、输出截断、审计记录

### 2.6 可观测与回放

- 会话日志
- 任务执行日志
- 工具调用日志
- 回放样本输出

## 3. 技术栈与选型

### 3.1 开发语言

建议：Python 3.12+

原因：

- 非常适合 Agent、工具编排、文件系统操作和子进程控制。
- AI 生态成熟，模型接入和实验验证效率更高。
- 对本地自动化、脚本执行、数据处理和插件式能力封装更直接。

### 3.2 CLI 框架

建议：`typer`

用途：

- 定义命令入口
- 解析参数
- 管理子命令

### 3.3 模型接入

建议：

- 定义统一 `LLMProvider` 接口
- 先接一个主模型供应商

要求：

- 支持结构化输出
- 支持工具调用
- 支持流式输出

### 3.4 大模型编排框架

主推荐：`PydanticAI`

原因：

- 与 Python 技术栈天然契合，适合做强类型 Agent Client。
- 结构化输出、工具调用、上下文组织能力够用，不会引入过重框架负担。
- 更适合和自研 `Skill Loader / Resolver`、`Tool Registry`、`Planner / Executor` 组合。
- 对当前项目这种“Client + Skills + Sandbox”架构更贴合。

使用建议：

- 把 `PydanticAI` 作为模型交互和工具调用编排层。
- 不把 Skill 机制直接绑死在框架内部，而是在 Client Runtime 里自行实现 Skill 解析和注入。
- 保留 `LLMProvider` 抽象，避免后续替换模型供应商时侵入业务层。

备选：`LangGraph`

适用场景：

- 需要显式状态图编排。
- 需要复杂分支、循环、人工审批节点。
- 需要更强的执行状态持久化和图级控制。

当前结论：

- 首期不选 `LangGraph` 作为主编排框架。
- 当任务流从“单 Agent + Skills”演进为“复杂图编排”时，再评估引入。

### 3.5 配置与校验

建议：

- `pydantic`
- `python-dotenv`
- JSON Schema

用途：

- Skill Manifest 校验
- 配置文件校验
- 工具参数校验

### 3.6 沙盒方案

推荐采用分阶段方案。

#### Phase 1：本地受控进程沙盒

主方案：

- `subprocess`
- `pathlib`
- `asyncio` 仅在需要异步工具执行时引入

适用目标：

- 本地开发
- 单机 Client 验证
- 可信工作区内的受控执行

应实现能力：

- 工作目录白名单
- 命令白名单或策略拦截
- 超时终止
- 输出截断
- 文件变更记录
- 审计日志

#### Phase 2：容器沙盒

主方案：

- Docker

推荐配置：

- rootless 模式
- 非 root 用户执行
- seccomp 配置
- 最小目录挂载
- 网络权限默认关闭或按需开启

适用目标：

- 更高安全要求
- 不可信任务输入
- 更严格的隔离场景

#### 远程沙盒备选

备选方案：

- `E2B`

适用场景：

- 需要托管型远程执行环境
- 不希望自行维护容器编排和隔离基础设施

当前结论：

- MVP 先做本地受控进程沙盒。
- 后续生产化优先演进到 Docker 容器沙盒。
- 如需云端隔离执行，再评估引入 E2B。

### 3.7 Shell 与文件执行基础能力

建议：

- `subprocess`
- `pathlib`
- `asyncio` 仅在需要异步工具执行时引入

说明：

- MVP 先做本地受控进程沙盒。
- 后续若安全要求提高，再接容器隔离。

### 3.8 日志

建议：

- `structlog` 或标准库 `logging`

### 3.9 测试

建议：

- `pytest`
- `pytest-asyncio` 在存在异步执行链路时使用
- `httpx` 仅在引入本地 HTTP 接口时使用

## 4. 组件拆解

### 4.1 CLI Shell

职责：

- 解析命令
- 创建会话
- 接收用户任务
- 输出执行结果

建议目录：

- `src/uni_agent/cli`

### 4.2 Agent Runtime

职责：

- 管理任务执行生命周期
- 驱动 Planner 和 Executor
- 处理错误、重试和终止
- **多轮重规划**：步骤失败后在 `prior_context` 下再次调用 Planner，直至本轮批次全成功或达到 `orchestrator_max_failed_rounds`
- **流式事件**：可选 `stream_event` 回调输出 NDJSON 语义事件（规划轮次、每步完成、结论、结束）
- **运行结论**：`TaskResult.conclusion`为确定性摘要 + 可选 LLM 合成（`run_conclusion.py`）
- **规划与轻量「意图」**：无独立分类器；**Skill** 由 `SkillMatcher` 对 `triggers` 做子串打分；**工具步骤** 由 `HeuristicPlanner`（正则与关键词优先级）或 `PydanticAIPlanner`（结构化 LLM 计划，失败/空计划回退启发式）生成。二者均在 **`HeuristicPlanner._memory_search_query`** 上达成一致：显式「memory search … / 搜索记忆：…」及 **回忆身份类短语**（如 `我是谁`）命中时 **优先单步 `memory_search`**；`PydanticAIPlanner` 在调用规划 LLM 前执行同一短路，减少「闲聊 → shell_exec+echo」误判

建议目录：

- `src/uni_agent/agent`

子组件：

- `orchestrator`
- `planner`
- `executor`
- `run_conclusion`（结论文本）
- `policy`（预留）

### 4.3 Skill System

职责：

- 扫描 Skills 目录（**仅** `skills_dir` 下**一层子目录**；每个目录至多一个 skill）
- 解析 Manifest：**`SKILL.md`**（YAML frontmatter + Markdown 正文，与 Codex / Claude Code / Cursor 习惯一致）或旧版 **`skill.yaml`** + `entry` 指向的 Markdown
- 发现可选资源：`references/*.md`、`references/*/*.md`、`reference.md`、`examples.md`；`scripts/` 下可执行文件路径写入 `instruction_text` 供规划器引用
- 将正文与小体积参考文件合并为 **`SkillSpec.instruction_text`**；**`PydanticAIPlanner`** 在用户提示中注入「Selected skill instructions」
- 匹配适用 Skill：`triggers` 子串加分；**无 triggers 时**用 `description` 拆词匹配（`skills/bundle.description_match_tokens`）
- 解析 Skill 元数据：`allowed_tools` / `required_tools`（**规划器不再用 `allowed_tools` 收窄工具集**；内置工具始终对 agent 可见；字段可保留作文档或后续策略）

建议目录：

- `src/uni_agent/skills`

子组件：

- `loader`（`SkillLoader(skills_dir, workspace)`，`workspace` 用于生成相对工作区的 `file_read` 路径提示）
- `bundle`（`SKILL.md` 解析、资源发现、`instruction_text` 合成）
- `matcher`
- `resolver`（预留）
- `validator`（预留；当前为 Pydantic `SkillSpec` 校验）

用户向 skill 目录与 `SKILL.md` 约定见：[Skills目录与SKILL说明.md](./Skills目录与SKILL说明.md)。

### 4.4 Tool System

职责：

- 注册工具
- 暴露工具 schema（**Orchestrator 每轮传给规划器的工具列表为注册表全量**；不因 Skill 的 `allowed_tools` 收窄）
- 分发执行请求

建议目录：

- `src/uni_agent/tools`

首批工具建议：

- `shell_exec`
- `file_read`
- `file_write`
- `http_fetch`
- `search_workspace`
- `memory_search`（本地 L0/L1 记忆检索；可选 LLM 关键词与合成）
- `delegate_task`（单层子 agent：内部 `build_orchestrator(enable_delegate_tool=False, ...)` 再 `run`；`TaskResult.parent_run_id`；子流式事件带 `delegation`；见 `uni_agent/bootstrap.py`）
- `command_lookup`（`shutil.which` 解析 PATH 上的命令；可选抓取 `--help`/`-h`；或按前缀列举可执行文件名）
- `run_python`（临时 `.py` 写入 `.uni-agent/code_run/`、`python3`/`python` 执行、cwd=workspace）

### 4.5 Sandbox

职责：

- 执行 shell（**argv 列表**，无 shell；首参数为可执行文件名）
- 限制工作目录
- 控制文件访问
- 记录审计信息
- **命令白名单**：默认见 `config.settings.DEFAULT_SANDBOX_ALLOWED_COMMANDS`（含 `du`；含 `python3`/`python` 以支持内置 `run_python`）
- **交互式批准**：非白名单命令在 TTY 下可提示用户一次批准（`sandbox_prompt_for_disallowed`）；非 TTY 默认拒绝

建议目录：

- `src/uni_agent/sandbox`

### 4.6 Context

职责：

- 管理会话上下文
- 注入 Skill 上下文
- 控制 Prompt 体积

实现说明（当前仓库）：

- **交互会话摘要**：`src/uni_agent/observability/client_session.py` — `compress_task_result_for_session` 将单次 `TaskResult` 压成有界单行摘要；`build_session_context_for_planner` 拼接最近若干条写入规划器的 `session_context`（条数/总字符上限见模块内常量）
- **单次 run 内重规划**：`src/uni_agent/agent/orchestrator.py` — `_format_prior_context` 将已执行步骤（含截断后的 `output`）提供给下一轮 `create_plan` 的 `prior_context`
- **LLM 系统提示**：`src/uni_agent/agent/system_prompts.py` 提供规划器/结论默认文案与 `UNI_AGENT_*` 覆盖逻辑（见下表）

建议目录（可选扩展）：

- `src/uni_agent/context`（若引入独立上下文模块时可迁入；当前逻辑主要在 `observability` / `agent`）

### 4.7 Observability

职责：

- 记录日志
- 输出 trace
- 存储回放数据
- 任务结果落盘至 `task_log_dir`（默认 `.uni-agent/runs/`，**已由 `.gitignore` 忽略，不提交仓库**）

建议目录：

- `src/uni_agent/observability`

### 4.8 主要环境变量（`UNI_AGENT_*`）

| 变量 | 含义 |
|------|------|
| `PLANNER_BACKEND` | `auto` / `heuristic` / `pydantic_ai` |
| `MODEL_NAME` | 模型 id（如 `openai:...`） |
| `OPENAI_BASE_URL` / `OPENAI_API_KEY` | OpenAI 兼容网关与密钥 |
| `SANDBOX_ALLOWED_COMMANDS` | 逗号分隔命令白名单 |
| `SANDBOX_PROMPT_FOR_DISALLOWED` | 是否对非白名单命令 TTY 询问批准 |
| `SANDBOX_COMMAND_TIMEOUT_SECONDS` | 子进程超时 |
| `ORCHESTRATOR_MAX_FAILED_ROUNDS` | 规划失败/空计划累计轮数上限 |
| `TOOL_STEP_RETRIES` | 单步工具失败重试次数 |
| `RUN_CONCLUSION_LLM` | 是否用 LLM 生成最终结论文本 |
| `GLOBAL_SYSTEM_PROMPT` | 可选；加在规划器与结论 LLM 的系统提示最前 |
| `PLANNER_INSTRUCTIONS` | 可选；整段替换规划器默认系统提示（再套全局前缀） |
| `CONCLUSION_SYSTEM_PROMPT` | 可选；整段替换结论合成默认系统提示（再套全局前缀） |
| `LLM_TEMPERATURE` / `LLM_RETRIES` | 规划器模型温度与 PydanticAI 输出/工具重试次数 |
| `WORKSPACE` | 工作区根路径；**未设置 `MEMORY_DIR` 时**记忆默认落在 **`<WORKSPACE>/.uni-agent/memory`**；**相对 `MEMORY_DIR`** 亦按此路径解析（不按进程 cwd） |
| `MEMORY_DIR` / `MEMORY_EXTRACT_ENABLED` / `MEMORY_IDLE_EXTRACT_SECONDS` | 本地记忆目录覆盖项、`uni-agent client` 空闲落盘；**`memory_index.json` + `l1/`** 布局见上 |
| `MEMORY_SEARCH_USE_LLM` / `MEMORY_SEARCH_MAX_HITS` | `memory_search`：为 true 且 LLM 可用时「关键词规划 → L0 匹配 → L1 合成回答」（关键词步骤失败时仍并入用户原 `query`）；否则字面子串检索 L0 |
| `DELEGATE_MAX_FAILED_ROUNDS` | 子 run 重规划失败轮上限；未设置则与 `ORCHESTRATOR_MAX_FAILED_ROUNDS` 相同 |
| `DELEGATE_TOOL_PROFILE` | `full`（默认）或 `readonly`（子 run 仅 `file_read` / `search_workspace` / `memory_search` / `command_lookup`） |
| `HTTP_FETCH_*` | 见 `settings.py`（字节上限、私网、域名白名单等） |

CLI 组装入口为 `uni_agent.bootstrap.build_orchestrator`（`cli/main.py` 与交互客户端均调用）。

CLI：`uni-agent run "<task>"` 默认 **`--stream`**（stderr NDJSON），`--no-stream` 关闭；stdout 仍为最终 JSON。

## 6. 核心实现步骤

### Step 1：初始化 Client 工程

完成项：

- 初始化 Python 项目
- 接入 CLI 框架
- 建立配置模块、日志模块和目录结构

产出：

- `uni-agent` CLI 可启动

### Step 2：定义核心数据结构

完成项：

- 定义 `Session`
- 定义 `Task`
- 定义 `PlanStep`
- 定义 `SkillSpec`
- 定义 `ToolSpec`

产出：

- Runtime 的核心类型稳定下来

### Step 3：实现 Skill System

完成项：

- 扫描本地 `skills/`
- 解析 `skill.yaml`
- 校验 Skill Manifest
- 实现匹配和优先级规则

产出：

- Client 能发现并加载 Skills

### Step 4：实现 Tool Registry

完成项：

- 定义工具注册器
- 为模型暴露工具描述
- 实现基础工具执行适配

产出：

- Agent 可通过统一接口调用工具

### Step 5：实现 Sandbox

完成项：

- 受控 shell 执行
- 工作目录限制
- 文件读写保护
- 超时控制
- 输出截断

产出：

- 工具执行具备最小安全边界

### Step 6：实现 Agent Runtime

完成项：

- 实现 Planner
- 实现 Executor
- 支持 Plan -> Act -> Observe 循环
- 支持失败回传和局部重规划

产出：

- Client 可以执行真实任务

### Step 7：实现 CLI 任务流

完成项：

- `uni-agent run "<task>"`（`--stream` / `--no-stream`，`--plan` 静态计划）
- `uni-agent skills`（列出已加载 skill 的 name / version / description）
- `uni-agent replay <run_id>`（`--format full|steps|jsonl`，`--verbose` 与 `full` 联用打印步骤）
- `uni-agent client`（交互 REPL，`--session` / `-s` 续会话）

产出：

- 用户可以直接通过命令使用 Client

### Step 8：补齐日志与回放

完成项：

- 任务日志
- 工具调用日志
- Skill 命中记录
- 结果回放文件

产出：

- 行为可追踪，可定位错误
