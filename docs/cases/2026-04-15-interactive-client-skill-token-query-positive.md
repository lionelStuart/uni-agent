# 正向案例：交互客户端 + 本地 Skill + 沙箱批准执行脚本

Skill 目录树、`SKILL.md` frontmatter 与正文约定见：[Skills 目录与 SKILL 说明](../Skills目录与SKILL说明.md)。

记录日期：2026-04-15（内容已脱敏，不含真实域名、用户标识与精确用量数值。）

## 场景

在仓库根目录启动交互客户端（`agent client`），工作区为某本地项目目录。用户已配置 OpenAI 兼容网关以使用结构化规划（LLM）。

用户连续发起三轮对话：

1. **「你有什么技能」** — 期望了解代理能力边界。
2. **「你能使用某某 skill 吗」** — 该 skill 为本地 `skills/` 下通过文件名匹配的插件（本例为 token 用量相关 skill）。
3. **「查询某用户上周的 token 消耗」** — 期望真正调用 skill 附带的查询脚本，而非仅 `echo` 说明。

## 观察到的正向行为

### 1. 能力说明（首轮）

- 规划器为「闲聊 / 能力说明」类问题生成了单步 `shell_exec` + `echo`，列出内置工具能力（读写、搜索、命令、网络拉取、记忆检索等）。
- `selected_skills` 为空（未命中特定 skill 触发词属预期或可接受）。

### 2. Skill 可用性（次轮）

- 任务包含 skill 名称关键词，**SkillMatcher** 命中对应 skill，`selected_skills` 含该 skill 名称。
- 规划仍为「口头确认可用」型一步计划；输出通过 `echo` 概括 skill 用途（查询用量与配额类能力），与 skill 文档描述一致。

### 3. 真实查询（第三轮）

- 仍选中同一 skill。
- 规划生成 **单步 `shell_exec`**，命令为 **`bash` + 工作区内脚本路径**（`skills/<skill-dir>/scripts/*.sh`）及 `--consumer`、时间窗参数。
- **沙箱**：`bash` 不在默认白名单，触发 **TTY 一次性批准**（`Allow this command once? [y/N]`）；用户输入 `y` 后命令执行成功。
- 脚本输出为 **JSON**（consumer、cluster、时间窗、按模型拆分行、汇总等），后续结论合成能正确摘要总量与按模型分布。

## 可复用的结论

- 本地 **SKILL.md + scripts/** 路径可被 LLM 规划为可执行命令；**workspace 相对路径** 与沙箱 `cwd=workspace` 一致时行为正确。
- 依赖 **`bash` / `zsh` / `python3` / `jq` / `curl`** 等时，需 **扩大白名单** 或接受 **交互批准**；非 TTY 场景下应提前配置白名单以免静默失败。
- 「有什么技能」与「能否用某 skill」易落在 **`echo` 捷径**；真正数据类任务需用户表述清晰或 skill 触发足够强，使规划走到脚本路径。
